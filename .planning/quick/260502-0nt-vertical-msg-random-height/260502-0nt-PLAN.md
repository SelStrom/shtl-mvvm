<diagnosis>
Framework-side всё уже готово:
- `ReactiveVirtualList<T>` имеет конструктор `(Func<int, float> heightProvider)`.
- `VirtualCollectionBinding.PositionView` (строка 318–328) выставляет `rt.sizeDelta` именно из `GetItemHeight(index)`, поэтому per-row высота применяется без правок prefab'а.
- Биндинг + `LayoutCalculator` корректно работают с переменной высотой.

Что не так в Sample сейчас:
- `ChatMessagesViewModel` экспонирует только два конструктора: `()` (через `:this(80f)`) и `(float fixedHeight)`.
- `VirtualListEntryScreen` создаёт vertical VM как `new ChatMessagesViewModel(80f)` — фиксированная высота.
- `VirtualListSampleWidget.OnMessageAdded` просто пушит ChatMessageViewModel в обе vm-коллекции, ничего не зная про высоту.

Что меняем (минимум):
1. Добавить третий конструктор `ChatMessagesViewModel(Func<int, float> heightProvider)`. Параметрless и float-fixed конструкторы НЕ трогаем (см. W-05 комментарий в файле — пере-присваивать поле `Messages` после конструктора нельзя).
2. В `VirtualListSampleWidget` завести приватный `List<float> _verticalHeights`, публичный метод `GetVerticalHeight(int index) => _verticalHeights[index]`, и в `OnMessageAdded` сначала `_verticalHeights.Add(UnityEngine.Random.Range(MinH, MaxH))`, а потом уже `_verticalMessagesVm.Messages.Add(...)`. Порядок важен: к моменту, когда `ReactiveVirtualList.GetItemHeight(newIndex)` вызывается биндингом, элемент уже на индексе и нужная высота уже в списке.
3. В `VirtualListEntryScreen.Start` сначала инстанцировать widget, потом передать его `GetVerticalHeight` в конструктор vertical VM. Horizontal оставляем `208f`.

Range:
- `MinHeight = 50f`, `MaxHeight = 150f`. Симметрично вокруг старого 80f, верхняя граница не сильно выше — prefab разумно тянется. Константы — `private const float MinHeight = 50f; private const float MaxHeight = 150f;` в `VirtualListSampleWidget` (там, где живут `_verticalHeights` и логика add).
- `UnityEngine.Random.Range(float, float)` — verbatim, без обёртки.

Stability контракт height-provider'а:
- `GetItemHeight(index)` вызывается каждый раз при `PositionView` для видимых элементов. Если генерировать `Random.Range` внутри провайдера — высота прыгает каждый кадр. Поэтому генерация ЕДИНОЖДЫ при добавлении, кэш в `_verticalHeights[index]`. Этого правила достаточно для сцены, в которой сообщения только добавляются — операций remove/clear в `VirtualListSampleWidget` нет. Если позже добавится remove — `_verticalHeights.RemoveAt(index)` нужно держать в синхроне (вне scope этого тика).

Что НЕ трогаем:
- `ChatMessageView`, `ChatMessageViewModel` — модель сообщения не знает о высоте.
- `ChatModel`, `ChatMessageModel` — `Height` в модели не появляется (это View-concern).
- Horizontal-список — остаётся fixed 208f.
- Существующие конструкторы `ChatMessagesViewModel()` / `(float)` — нужны для prefab-инстанцирования через UnityInspector (`new()` constraint в `AbstractWidgetView<TViewModel>`), а также для horizontal-сценария.
- Никаких изменений в Runtime — задача чисто sample-side.
</diagnosis>
