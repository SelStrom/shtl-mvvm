# Quick Task 260502-05g: Drag Vertical Direction Inverted — Summary

Reversed-scroll bug in vertical drag fixed: dragging the finger downward now scrolls the content downward (natural-scroll convention), matching the wheel direction and the `VirtualCollectionBinding.PositionView` mapping.

## Root Cause

`VirtualScrollRect.GetLocalPosition` returned the raw `localPoint.y` for the Vertical axis. Unity UI local-y grows upward, while `_scrollPosition` grows from the start of the content (top → bottom). The OnDrag math (`_scrollPosition -= delta`) is axis-symmetric and works correctly for Horizontal because local-x and `_scrollPosition` already share a compatible direction-of-travel. For Vertical the y-sign needed flipping so a finger moving down produces a positive `pos` value (same direction-of-travel as a finger moving right horizontally).

## What Changed

### Sign fix (commit `0965e96`)

`Runtime/Core/VirtualScroll/VirtualScrollRect.cs`, `GetLocalPosition`, line 454.

Before:
```csharp
return _axis == ScrollAxis.Vertical ? localPoint.y : localPoint.x;
```

After:
```csharp
// Vertical: invert Y so a finger moving DOWN produces a positive delta in the
// same direction-of-travel convention as Horizontal "finger right". Unity UI Y
// grows upward, while scrollPosition grows away from the start of the content
// (top → bottom for Vertical, left → right for Horizontal), so the local-Y sign
// must be flipped to keep the OnDrag math (`_scrollPosition -= delta`) symmetric
// across both axes and to keep _velocity sign consistent for inertia in LateUpdate.
return _axis == ScrollAxis.Vertical ? -localPoint.y : localPoint.x;
```

Nothing else in the file changed — `OnDrag`, `OnBeginDrag`, `OnEndDrag`, `OnScroll`, `LateUpdate`, `RubberDelta` path, elastic / inertia / scrollbar logic and the public/internal API surface are untouched.

### Regression tests (commit `23f2703`)

`Tests/Editor/VirtualScrollRectDragTests.cs` (new) — 4 EditMode tests, mirroring the reflection-based style of `VirtualScrollRectWheelTests`:

| Test | Scenario | Assertion |
|------|----------|-----------|
| `OnDrag_VerticalFingerDown_DecreasesScrollPosition` | `OnBeginDrag(localY=0)` → `OnDrag(localY=-50)` | `ScrollPosition < startPos` |
| `OnDrag_VerticalFingerUp_IncreasesScrollPosition`   | `OnBeginDrag(localY=0)` → `OnDrag(localY=50)`  | `ScrollPosition > startPos` |
| `OnDrag_VerticalFingerDown_VelocitySignMatchesDelta` | finger DOWN | `Velocity < 0` |
| `OnDrag_VerticalFingerUp_VelocitySignMatchesDelta`   | finger UP   | `Velocity > 0` |

Tests use `MovementType.Clamped`, viewport size 400×300 at world origin with centred pivot, and `pressEventCamera = null` so `MakeDragEvent(localY)` is deterministic. RED-gate property: reverting the GetLocalPosition sign-flip flips every assertion and all four tests fail.

`Tests/Editor/VirtualScrollRectDragTests.cs.meta` (new) — Unity meta file with stable GUID `25555c56bbd84767a2f2818f0b6f28fb`.

## Path Not Taken

The executor agent originally proposed flipping the sign inside `OnDrag` (`_scrollPosition += delta; _velocity = delta / dt`). That fix passes the same vertical regression tests, but it would have **broken Horizontal** (which already follows natural-scroll: finger right → content right via `_scrollPosition -= delta` with positive delta from local-x). The orchestrator caught this asymmetry by checking `VirtualCollectionBinding.PositionView` for both axes and tracing through Horizontal sign math; the executor's fix was discarded and the alternative single-line fix in `GetLocalPosition` was used instead. The user had also independently arrived at the same `-localPoint.y` fix as a local working-tree change before the orchestrator started — the working-tree change is what got committed.
