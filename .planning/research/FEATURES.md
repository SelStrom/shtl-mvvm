# Исследование фич

**Домен:** MVVM-фреймворк для Unity (data binding, виртуализированные списки)
**Дата исследования:** 2026-04-09
**Уверенность:** MEDIUM-HIGH

## Ландшафт фич

### Table Stakes (пользователи ожидают по умолчанию)

Фичи, которые пользователи считают само собой разумеющимися. Их отсутствие = фреймворк воспринимается как недоделанный.

| Фича | Почему ожидается | Сложность | Примечания |
|------|------------------|-----------|------------|
| Режимы привязки: OneWay, TwoWay, OneTime | Стандарт MVVM (WPF, MAUI, Loxodon, Aspid.MVVM) — все конкуренты поддерживают минимум 3 режима | MEDIUM | Сейчас Shtl.Mvvm поддерживает только OneWay. TwoWay и OneTime — минимальный ожидаемый набор |
| Two-way биндинг для input-полей | InputField, Slider, Toggle — пользователь вводит данные, ожидает синхронизацию с ViewModel | MEDIUM | Нужны extension-методы `To()` для InputField, Slider, Toggle с обратной подпиской на `onValueChanged` |
| Конвертеры значений (Value Converters) | WPF/MAUI IValueConverter — стандартный паттерн. Нужен для трансформации данных между ViewModel и View | LOW | Простой интерфейс `IValueConverter<TSource, TTarget>` + chain-метод `.WithConverter()` в билдере |
| Ленивая/отложенная активация биндинга | Билдер-паттерн подразумевает конфигурирование до активации. Все зрелые фреймворки отделяют создание от подписки | MEDIUM | Текущий `BindFrom.LinkTo()` активирует немедленно. Нужен промежуточный этап конфигурации |
| Виртуализированный список: recycling элементов | 50+ элементов без виртуализации = просадка FPS. Unity UI Toolkit ListView уже виртуализирует. В uGUI — ожидается от фреймворка | HIGH | Основная фича милестоуна. Pool элементов + viewport culling |
| Виртуализированный список: вертикальный скролл | Базовый режим прокрутки, используется в 90% случаев | HIGH | Первый реализуемый layout |
| Автоматический Dispose при уничтожении GameObject | Утечки подписок — главная боль Unity MVVM. Пользователь ожидает, что биндинги очищаются автоматически | LOW | Уже реализовано через AbstractWidgetView. Подтвердить работу с виртуализированным списком |

### Differentiators (конкурентное преимущество)

Фичи, которые выделяют продукт. Не обязательны, но ценны.

| Фича | Ценность | Сложность | Примечания |
|------|----------|-----------|------------|
| Fluent builder-паттерн биндингов с chain-методами | Лаконичный API: `Bind.From(x).WithConverter(c).TwoWay().To(y)` — ни один Unity MVVM так не делает. Все используют атрибуты или конфиг-объекты | MEDIUM | Ключевой дифференциатор Shtl.Mvvm. Builder-структура с ленивым созданием |
| Кэширование ранее собранных биндингов | Повторное открытие UI-экрана не пересоздает биндинги, а переиспользует готовые. Экономия аллокаций | MEDIUM | BindingPool уже есть для самих биндингов, нужно кэширование конфигурации билдера |
| Виртуализированный список: горизонтальный скролл | Карусели, горизонтальные галереи — частый паттерн в мобильных играх | MEDIUM | Переиспользование логики вертикального с заменой оси |
| Виртуализированный список: grid-layout | Инвентари, магазины, каталоги — сетки очень распространены в геймдеве | HIGH | Значительно сложнее линейного списка. Нужен расчет строк/столбцов + recycling в двух измерениях |
| Виртуализированный список: элементы переменной высоты | Чаты, ленты новостей — элементы разной высоты | HIGH | Требует измерения высоты после рендера, кумулятивный расчет позиций, binary search для индексов |
| Overscan-буфер (рендер элементов за пределами viewport) | Устранение мерцания при быстром скролле | LOW | Стандартный прием: рендерить N дополнительных элементов сверху и снизу |
| Zero-alloc hot path (пулирование + структуры) | Unity-специфика: GC spikes = подтормаживания. Loxodon заявляет zero GC — это конкурентный аргумент | MEDIUM | BindingPool уже есть. Обеспечить zero-alloc в цикле обновления виртуализированного списка |
| Snap-to-item прокрутка | Пагинация, карусели — элемент всегда по центру после отпускания скролла | MEDIUM | Доп. компонент поверх ScrollRect. Не блокирует основную виртуализацию |

### Anti-Features (намеренно НЕ делаем)

Фичи, которые кажутся хорошими, но создают проблемы.

| Фича | Почему запрашивают | Почему проблемно | Альтернатива |
|------|-------------------|------------------|--------------|
| Рефлексия для автобиндинга (атрибуты `[Bind]` на полях) | Удобно: не писать код биндингов вручную | Производительность (рефлексия = аллокации), магическое поведение, сложная отладка, несовместимо с IL2CPP stripping | Явный fluent API — `Bind.From().To()`. Код написать один раз, зато читаемый и предсказуемый |
| XAML/XML-разметка для описания биндингов | WPF-подход: биндинги в разметке | Unity не поддерживает нативно. Создает параллельную систему описания UI. Сложная интеграция с uGUI | Биндинги в C#-коде через fluent API |
| ICommand / AsyncCommand паттерн | Стандарт WPF/MAUI MVVM для команд | Overkill для Unity — кнопки уже привязываются через `Bind.From(button).To(action)`. ICommand добавляет абстракцию без пользы в контексте Unity | `ReactiveValue<Action>` + `ButtonEventBinding` уже покрывают use case |
| DI-интеграция / сервис-локатор | Удобно инжектить зависимости в ViewModel | Нарушает независимость пакета. Привязка к VContainer/Zenject/etc. | Widget-паттерн: зависимости передаются явно через конструктор/параметры |
| UIToolkit поддержка | Unity 6 продвигает UIToolkit | Полная смена парадигмы рендеринга UI. Несовместимо с uGUI архитектурой | Отдельный пакет/модуль в будущем, если будет спрос |
| Анимация в биндингах (Tween при изменении значения) | Красивые переходы | Смешивает ответственности (binding != animation). Сложность кода, конфликты с существующими анимациями | ReactiveAwaitable уже покрывает кейс ожидания анимации. Анимация — задача View-слоя |
| Бесконечный скролл с подгрузкой данных | Ленты, пагинация с сервера | Смешивает UI-виртуализацию с бизнес-логикой загрузки данных | ViewModel управляет данными в ReactiveList, View виртуализирует. Подгрузка — ответственность ViewModel |

## Зависимости между фичами

```
[Ленивая активация биндинга (Builder)]
    +-- требуется для --> [TwoWay биндинг]
    +-- требуется для --> [Конвертеры значений]
    +-- требуется для --> [Кэширование конфигурации билдера]

[TwoWay биндинг]
    +-- требуется для --> [Two-way InputField/Slider/Toggle]

[Виртуализированный список: вертикальный]
    +-- требуется для --> [Горизонтальный скролл]
    +-- требуется для --> [Grid-layout]
    +-- требуется для --> [Элементы переменной высоты]
    +-- улучшается --> [Overscan-буфер]
    +-- улучшается --> [Snap-to-item]
    +-- улучшается --> [Zero-alloc hot path]

[Recycling элементов]
    +-- требуется для --> [Виртуализированный список: вертикальный]

[BindingPool (существует)]
    +-- используется --> [Recycling элементов]
```

### Пояснения к зависимостям

- **Builder требуется для TwoWay**: двусторонний биндинг — это конфигурация, которая должна быть задана до активации подписки. Без промежуточного этапа (builder) невозможно отделить "направление" от "создания".
- **Builder требуется для Value Converters**: конвертер — часть конфигурации биндинга, применяется при chain-методе `.WithConverter()` до финального `.To()`.
- **Вертикальный список требуется для горизонтального**: архитектура одна, меняется только ось. Горизонтальный — параметризация вертикального.
- **Вертикальный список требуется для Grid**: Grid = несколько столбцов поверх вертикальной прокрутки. Зависит от базовой инфраструктуры recycling.

## Определение MVP (для текущего милестоуна)

### Обязательно (v1.2)

Минимальный набор для выпуска милестоуна.

- [ ] **Builder-структура биндингов** — основа для конфигурирования TwoWay и конвертеров. Обратная совместимость: `Bind.From(x).To(y)` работает как раньше
- [ ] **TwoWay биндинг для InputField, Slider, Toggle** — самый запрашиваемый кейс, без него two-way бессмысленен
- [ ] **Виртуализированный вертикальный список** — recycling, viewport culling, фиксированный размер элементов
- [ ] **Overscan-буфер** — без него быстрый скролл мерцает, это не опционально

### Добавить после валидации (v1.2.x)

- [ ] **Конвертеры значений** — когда builder API стабилизирован и есть реальные use cases
- [ ] **Горизонтальный скролл** — параметризация вертикального, низкий риск
- [ ] **Snap-to-item** — после стабилизации базового скролла
- [ ] **OneTime режим привязки** — простой, но требует стабильного builder API

### На будущее (v1.3+)

- [ ] **Grid-layout** — существенная сложность, отложить до реального запроса
- [ ] **Элементы переменной высоты** — требует значительных архитектурных решений (измерение, кумулятивные позиции, binary search)
- [ ] **Кэширование конфигурации билдера** — оптимизация, не блокирующая функциональность

## Анализ конкурентов

| Фича | Loxodon Framework | UnityMvvmToolkit (LibraStack) | Aspid.MVVM | Unity-Weld | Shtl.Mvvm (план) |
|------|-------------------|-------------------------------|------------|------------|-------------------|
| Binding modes | OneWay, TwoWay + конвертеры | OneWay, TwoWay через UXML | OneWay, TwoWay, OneTime, OneWayToSource | OneWay, TwoWay | OneWay (есть), TwoWay, OneTime (план) |
| API стиль | C# код + атрибуты | UXML атрибуты + C# | Атрибуты + конфигурация | Атрибуты через Inspector | **Fluent builder (уникально)** |
| Целевая UI-система | uGUI + FairyGUI | UI Toolkit + uGUI | uGUI | uGUI | uGUI |
| Value Converters | Да | Да (IValueConverter) | Нет данных | Нет | Да (план) |
| Async Commands | Нет данных | Да (AsyncCommand + UniTask) | Нет данных | Нет | Нет (anti-feature) |
| Виртуализация списков | Нет (отдельно) | Нет (UI Toolkit нативно) | Нет данных | Нет | **Встроенная (план)** |
| Zero-alloc | Заявлен | Нет данных | Заявлен (без рефлексии) | Нет (рефлексия) | Да (BindingPool) |
| Object pooling | Нет данных | Нет данных | Нет данных | Нет | **Да (BindingPool + recycling)** |

**Ключевое отличие Shtl.Mvvm**: fluent builder API + встроенная виртуализация списков + zero-alloc через пулирование. Ни один конкурент не объединяет все три.

## Источники

- [Loxodon Framework (Unity-MVVM)](https://github.com/vovgou/loxodon-framework) — зрелый фреймворк с data binding и observable collections
- [UnityMvvmToolkit (LibraStack)](https://github.com/LibraStack/UnityMvvmToolkit) — UI Toolkit + uGUI, async commands, value converters
- [Aspid.MVVM](https://discussions.unity.com/t/released-aspid-mvvm/1684510) — 4 режима привязки без рефлексии

- [Unity 6 Data Binding](https://docs.unity3d.com/6000.3/Documentation/Manual/best-practice-guides/ui-toolkit-for-advanced-unity-developers/data-binding.html) — нативная привязка данных в UI Toolkit
- [UnityRecyclingListView](https://github.com/sinbad/UnityRecyclingListView) — recycling-список для uGUI
- [Recyclable Scroll Rect](https://assetstore.unity.com/packages/tools/gui/recyclable-scroll-rect-optimized-list-grid-view-178560) — коммерческое решение recycling для uGUI
- [.NET MAUI Binding Modes](https://learn.microsoft.com/en-us/dotnet/maui/fundamentals/data-binding/binding-mode) — стандарт режимов привязки в .NET
- [Виртуализация с переменной высотой (aspnetcore)](https://github.com/dotnet/aspnetcore/issues/65158) — design proposal для variable-height виртуализации
